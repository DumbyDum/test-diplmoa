# Метрики И Формулы OmniGuard 2.0

Ниже собраны все основные метрики, которые использует проект.

## 1. Метрики сравнения изображений

Эти метрики сравнивают текущее изображение с защищенной опорной версией.

### 1.1. MSE

**Формула:**

`MSE = (1 / N) * sum((I_ref - I_test)^2)`

Где:

- `I_ref` - опорное изображение;
- `I_test` - анализируемое изображение;
- `N` - число значений пикселей.

**Что показывает:** средний квадрат ошибки между изображениями.  
**Как интерпретировать:** чем меньше, тем изображения ближе.

### 1.2. MAE

**Формула:**

`MAE = (1 / N) * sum(|I_ref - I_test|)`

**Что показывает:** среднее абсолютное отличие пикселей.  
**Как интерпретировать:** чем меньше, тем лучше.

### 1.3. RMSE

**Формула:**

`RMSE = sqrt(MSE)`

**Что показывает:** корень из средней квадратичной ошибки.  
**Как интерпретировать:** удобен тем, что измеряется в той же шкале, что и пиксели.

### 1.4. PSNR

**Формула:**

`PSNR = 10 * log10((MAX^2) / MSE)`

Где `MAX = 255` для 8-битных изображений.

**Что показывает:** отношение полезного сигнала к шуму/искажению.  
**Как интерпретировать:** чем выше, тем ближе изображения.

### 1.5. SSIM

**Смысл формулы:** сравнивает яркость, контраст и структуру изображения.

**Что показывает:** насколько похожа структура изображений с точки зрения восприятия.  
**Диапазон:** обычно от `0` до `1`.  
**Как интерпретировать:** чем ближе к `1`, тем изображения структурно ближе.

### 1.6. Changed Pixel Ratio

**Формула:**

`ChangedPixelRatio = changed_pixels / all_pixels`

Пиксель считается измененным, если хотя бы один канал изменился сильнее заданного порога.

**Что показывает:** какая доля пикселей заметно изменилась по сравнению с опорной версией.  
**Как интерпретировать:** полезно для оценки масштаба правки.

## 2. Метрики payload

Эти метрики относятся к скрытой полезной нагрузке, встроенной в изображение.

### 2.1. Payload Bit Accuracy

**Формула:**

`BitAccuracy = correct_bits / all_bits`

**Что показывает:** какая доля бит payload была правильно восстановлена.  
**Диапазон:** от `0` до `1`.  
**Как интерпретировать:** чем ближе к `1`, тем устойчивее payload к искажению.

### 2.2. Payload Auth OK

Это не числовая метрика, а логический флаг.

**Что показывает:** прошла ли встроенная контрольная подпись payload проверку.  
**Как интерпретировать:** если `false`, встроенная полезная нагрузка была повреждена или изменилась слишком сильно.

### 2.3. Payload Document Match

Это логический флаг.

**Что показывает:** совпадает ли восстановленный hash `document_id` с ожидаемым `document_id`.  
**Как интерпретировать:** если `false`, изображение либо изменено, либо не соответствует ожидаемой записи.

## 3. Метрики локализации изменений

Эти метрики относятся к heatmap и бинарной маске.

### 3.1. Tamper Score Mean

**Формула:**

`TamperScoreMean = mean(H)`

Где `H` - heatmap аномалий.

**Что показывает:** средний уровень подозрительности по всему изображению.  
**Как интерпретировать:** полезен для оценки общего уровня искажения.

### 3.2. Tamper Score Max

**Формула:**

`TamperScoreMax = max(H)`

**Что показывает:** максимальную локальную аномалию.  
**Как интерпретировать:** особенно важен при небольших локальных правках.

### 3.3. Tamper Ratio

**Формула:**

`TamperRatio = mask_pixels / all_pixels`

Где `mask_pixels` - число пикселей, попавших в бинарную маску.

**Что показывает:** долю изображения, которую система считает подозрительной.  
**Как интерпретировать:** чем больше значение, тем больше измененная область или тем сильнее реакция локализатора.

## 4. Метрики масок

Эти метрики считаются только тогда, когда известна эталонная маска изменения.

### 4.1. Precision

**Формула:**

`Precision = TP / (TP + FP)`

**Что показывает:** какая доля пикселей, отмеченных системой как измененные, действительно изменена.  
**Как интерпретировать:** высокая precision означает мало ложных срабатываний.

### 4.2. Recall

**Формула:**

`Recall = TP / (TP + FN)`

**Что показывает:** какую долю реально измененных пикселей система смогла найти.  
**Как интерпретировать:** высокая recall означает, что система не пропускает изменения.

### 4.3. F1

**Формула:**

`F1 = 2 * Precision * Recall / (Precision + Recall)`

**Что показывает:** баланс между precision и recall.  
**Как интерпретировать:** полезна как единая сводная метрика качества маски.

### 4.4. IoU

**Формула:**

`IoU = Intersection / Union`

**Что показывает:** долю пересечения эталонной и предсказанной масок относительно их объединения.  
**Как интерпретировать:** чем выше, тем лучше совпадение масок.

### 4.5. Dice

**Формула:**

`Dice = 2 * Intersection / (|A| + |B|)`

Где:

- `A` - эталонная маска;
- `B` - предсказанная маска.

**Что показывает:** степень совпадения двух масок.  
**Как интерпретировать:** чем выше, тем лучше.

## 5. Какие метрики важнее в разных сценариях

### Если нужно проверить целостность изображения

Смотри в первую очередь:

- `payload_auth_ok`
- `payload_document_match`
- `payload_bit_accuracy`

### Если нужно понять, где именно было изменение

Смотри в первую очередь:

- `tamper_score_max`
- `tamper_ratio`
- `precision`
- `recall`
- `F1`
- `IoU`
- `Dice`

### Если нужно оценить общую степень визуального искажения

Смотри:

- `MSE`
- `MAE`
- `RMSE`
- `PSNR`
- `SSIM`
- `changed_pixel_ratio`

---

## 6. Как читать результаты без математики

Если совсем по-простому:

- `payload` отвечает за вопрос: **целы ли встроенные данные**
- `tamper heatmap` отвечает за вопрос: **где система видит подозрительные области**
- `mask`-метрики отвечают за вопрос: **насколько хорошо локализация совпала с реальной областью изменения**
- `image difference`-метрики отвечают за вопрос: **насколько сильно текущее изображение отличается от опорного**
