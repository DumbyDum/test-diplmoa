# Полное описание пайплайна, метрик и heatmap

Этот файл описывает внутреннюю логику OmniGuard 2.0: из каких компонентов состоит код, в какой последовательности работает система, какие метрики выводятся в UI и отчетах, как устроен payload и как строится heatmap.

## 1. Главная идея системы

OmniGuard работает с изображениями и решает несколько задач:

1. встроить в изображение цифровой водяной знак;
2. встроить проверяемый payload, связанный с `document_id`;
3. после изменения изображения попытаться извлечь payload обратно;
4. построить карту подозрительных областей `heatmap`;
5. превратить heatmap в бинарную маску;
6. посчитать метрики качества, емкости и устойчивости.

Система состоит из двух контуров:

- **основной OmniGuard-контур** — нейросетевой watermark, payload, анализ и heatmap;
- **экспериментальный контур требований** — LSB, DCT, атаки и метрики `PSNR`, `SSIM`, `bpp`, `BER`.

## 2. Основные файлы и их роль

`omniguard/ui.py` — интерфейс Gradio. Создает вкладки, кнопки, таблицы, раскрывающиеся технические отчеты и справочник.

`omniguard/service.py` — центральный движок `OmniGuardEngine`. В нем собраны методы `protect_image`, `edit_image`, `analyze_image`.

`omniguard/payload.py` — сборка, кодирование, декодирование и проверка payload.

`omniguard/legacy_models.py` — работа с исходными нейросетевыми моделями: встраивание tamper watermark, встраивание 100 бит payload, извлечение payload, восстановление карты нарушения watermark.

`omniguard/metrics.py` — все метрики: `MSE`, `MAE`, `RMSE`, `PSNR`, `SSIM`, `bpp`, `BER`, `bit_accuracy`, `IoU`, `Dice`, `Precision`, `Recall`, `F1`.

`omniguard/attacks.py` — атаки из требований: JPEG, Gaussian noise, Salt-and-Pepper, crop, яркость, median filter, downscale/upscale.

`omniguard/basic_watermarking.py` — базовые методы ЦВЗ: `LSB` и `DCT`.

`omniguard/requirement_experiments.py` — пакетный benchmark для большого набора изображений.

`omniguard/editing.py` — встроенные редакторы изображения: OpenCV и diffusers.

`omniguard/image_ops.py` — загрузка изображений, преобразование RGB/RGBA/grayscale, сохранение, resize, overlay heatmap.

`omniguard/schemas.py` — структуры данных отчетов: `ProtectionResult`, `AnalysisResult`, `AttackResult`, payload-структуры.

## 3. Пайплайн защиты изображения

Когда пользователь нажимает `Защитить изображение`, вызывается:

`OmniGuardEngine.protect_image(image, document_id)`

Последовательность:

1. `load_image(image)` загружает изображение и приводит его к RGB `uint8`.
2. `limit_image_side(...)` ограничивает слишком большие изображения.
3. `build_payload_bits(document_id, hmac_secret)` собирает payload.
4. `models.embed_tamper_watermark(source)` встраивает tamper-sensitive watermark.
5. `models.embed_payload_bits(tamper_protected, payload.encoded_bits)` встраивает 100 бит payload.
6. Возвращается `ProtectionResult`.

Итог: защищенное изображение содержит два слоя:

- скрытый watermark для локализации изменений;
- payload для проверки встроенных данных.

## 4. Структура payload

Payload — это не просто случайные биты. Это маленькая проверяемая запись.

Сырая структура payload занимает `56 бит`:

| Поле | Размер | Назначение |
|---|---:|---|
| `version` | 4 бита | версия формата payload |
| `issued_hours` | 20 бит | время создания в часах от `2026-01-01 00:00 UTC` |
| `document_hash` | 16 бит | короткий hash от `document_id` |
| `nonce` | 8 бит | случайное число для разнообразия payload |
| `auth_tag` | 8 бит | HMAC-тег для проверки целостности |

Формирование:

1. `document_id` кодируется в UTF-8.
2. Считается `SHA-256(document_id)`.
3. Из digest берется 16-битный `document_hash`.
4. Формируются первые 48 бит: `version + issued_hours + document_hash + nonce`.
5. По этим 48 битам считается HMAC-SHA256.
6. Первый байт HMAC становится `auth_tag`.
7. Получаются 56 raw bits.

Формула HMAC:

`auth_tag = first_8_bits(HMAC_SHA256(secret_key, base_bits))`

где `base_bits` — первые 48 бит payload.

## 5. Почему payload становится 100-битным

Нейросетевая payload-модель ожидает `100 бит`, поэтому 56 raw bits кодируются Hamming-кодом.

Используется Hamming `(7,4)`:

- каждые 4 бита данных превращаются в 7 бит;
- 56 бит = 14 групп по 4 бита;
- 14 групп * 7 бит = 98 бит;
- добавляются 2 pad-бита `[0, 1]`;
- итог = 100 бит.

Для четырех информационных бит:

`d1, d2, d3, d4`

считаются проверочные биты:

`p1 = d1 xor d2 xor d4`

`p2 = d1 xor d3 xor d4`

`p3 = d2 xor d3 xor d4`

Кодовое слово:

`[p1, p2, d1, p3, d2, d3, d4]`

При декодировании считается синдром:

`s1 = b1 xor b3 xor b5 xor b7`

`s2 = b2 xor b3 xor b6 xor b7`

`s3 = b4 xor b5 xor b6 xor b7`

`syndrome = s1 + 2*s2 + 4*s3`

Если `syndrome != 0`, значит в 7-битном слове найдена одиночная ошибка, и соответствующий бит переворачивается.

## 6. Что означает `document_id`

`document_id` — это внешний идентификатор изображения или документа.

Пример:

`document_id = contract-2026-001`

В payload не хранится сам текст `contract-2026-001`. Вместо этого хранится короткий 16-битный hash.

Зачем так сделано:

- payload очень маленький;
- нельзя положить длинный текст целиком;
- hash позволяет проверить, соответствует ли изображение ожидаемому идентификатору.

При анализе пользователь вводит ожидаемый `document_id`. Система снова считает его hash и сравнивает с hash, который извлекла из изображения.

Если hash совпал:

`payload_document_match = true`

Если не совпал:

`payload_document_match = false`

Если пользователь не ввел ожидаемый `document_id`:

`payload_document_match = null`

## 7. Пайплайн анализа изображения

Когда пользователь нажимает `Проанализировать`, вызывается:

`OmniGuardEngine.analyze_image(...)`

Последовательность:

1. Изображение загружается и приводится к RGB.
2. Payload-декодер извлекает 100 бит.
3. `decode_payload_bits(...)` декодирует Hamming, проверяет HMAC и `document_id`.
4. Модель восстанавливает карту нарушения watermark.
5. Если есть защищенная опорная версия, строится reference-map.
6. В зависимости от режима создается итоговая heatmap.
7. Heatmap превращается в бинарную маску.
8. Считаются метрики анализа.
9. В UI выводятся heatmap, mask, краткая интерпретация и технический отчет.

## 8. Режимы локализации

### 8.1. `watermark`

Используется только скрытый watermark.

Система пытается восстановить секретный watermark из анализируемого изображения и сравнивает его с ожидаемым секретным изображением.

Плюс: работает без опорной версии.

Минус: локализация может быть более шумной.

### 8.2. `reference`

Используется только сравнение текущего изображения с защищенной опорной версией.

Плюс: хорошо показывает физические отличия между двумя файлами.

Минус: нужна защищенная версия до редактирования.

### 8.3. `hybrid`

Использует оба источника:

- нарушение watermark;
- отличие от опорной версии.

Итоговая карта строится так:

`combined = 0.35 * watermark_map + 0.65 * reference_map`

Затем:

`combined = max(combined, 0.95 * reference_map)`

Это означает: если reference-map уверенно показывает изменение, итоговая heatmap не должна его потерять.

## 9. Как строится watermark heatmap

Watermark heatmap строится внутри `LegacyModelBundle.reveal_tamper_mask`.

Идея:

1. При защите в изображение был встроен секретный watermark.
2. При анализе модель запускается в обратном режиме.
3. Она пытается восстановить скрытый secret-image.
4. Если изображение не менялось, восстановленный secret похож на ожидаемый.
5. Если область менялась, восстановленный secret в этой области искажается.
6. Разность между восстановленным и ожидаемым secret превращается в карту аномалий.

В коде:

1. `source` переводится в tensor `[0, 1]`.
2. Выполняется DWT-преобразование.
3. Модель запускается с `rev=True`.
4. Из выхода берется `secret_rev`.
5. Выполняется обратное wavelet-преобразование `IWT`.
6. Готовится эталонный `reference_secret`.
7. Считается средняя абсолютная разница:

`residual_mask = mean(abs(artifact - reference_secret), channels)`

где:

- `artifact` — восстановленный secret;
- `reference_secret` — ожидаемый secret;
- `mean(..., channels)` усредняет разницу по каналам.

После этого карта масштабируется к размеру исходного изображения.

## 10. Нормализация heatmap

Сырые значения модели не всегда удобно читать напрямую. Поэтому применяется нормализация:

`normalized = (score - low) / (high - low)`

где:

- `low` — нижний перцентиль;
- `high` — верхний перцентиль.

Для watermark-map используются:

- `low_percentile = 80.0`;
- `high_percentile = 99.8`;
- `blur_sigma = 1.0`.

Это означает: слабый фон подавляется, а сильные локальные отклонения становятся ярче.

После нормализации:

`normalized = clip(normalized, 0, 1)`

Затем в UI карта переводится в `uint8`:

`heatmap_uint8 = heatmap_float * 255`

## 11. Как строится reference heatmap

Reference heatmap строится, если есть защищенная опорная версия.

Входы:

- `source` — текущее изображение;
- `reference` — защищенное изображение до внешнего изменения.

Шаги:

1. Оба изображения переводятся в `[0, 1]`.
2. Считается абсолютная разница по каналам:

`channel_diff = abs(source - reference)`

3. Считается grayscale-разница.
4. Считается максимальная разница по каналам:

`max_channel = max(channel_diff_R, channel_diff_G, channel_diff_B)`

5. Берется максимум:

`blended = max(gray_diff, max_channel)`

6. Карта нормализуется через перцентили:

- `low_percentile = 65.0`;
- `high_percentile = 99.5`;
- `blur_sigma = 1.2`.

Reference heatmap обычно лучше выделяет конкретные ручные правки, потому что она сравнивает две версии изображения напрямую.

## 12. Как heatmap превращается в бинарную маску

Heatmap — плавная карта. Бинарная маска — уже жесткое решение:

- `255` — подозрительный пиксель;
- `0` — нормальный пиксель.

Если reference-map нет:

`binary = heatmap >= threshold`

Если reference-map есть, используется более строгая логика:

1. Reference-map переводится в `uint8`.
2. По ней считается Otsu-порог.
3. Итоговый reference threshold:

`reference_threshold = max(user_threshold, otsu_threshold / 255)`

4. Строится direct-mask:

`direct_mask = reference_map >= max(0.08, reference_threshold * 0.7)`

5. Строится combined-mask:

`combined_mask = heatmap >= max(user_threshold, reference_threshold * 0.85)`

6. Итог:

`binary = direct_mask AND (combined_mask OR reference_map >= 0.22)`

После этого применяются морфологические операции:

- `MORPH_OPEN 3x3` убирает мелкий шум;
- `MORPH_CLOSE 7x7` закрывает маленькие дырки;
- `dilate 3x3` немного расширяет найденную область.

## 13. Что такое порог бинарной маски

Порог — это граница, выше которой пиксель считается подозрительным.

Низкий порог:

- выделяет больше областей;
- повышает чувствительность;
- может дать больше ложных срабатываний.

Высокий порог:

- выделяет меньше областей;
- делает маску строже;
- может пропустить слабые изменения.

В коде значение по умолчанию:

`tamper_mask_threshold = 0.03`

## 14. Метрики payload в анализе

### `payload_auth_ok`

Тип: boolean.

Показывает, прошла ли HMAC-проверка payload.

Логика:

1. Из изображения извлекаются 100 бит.
2. Hamming-декодер восстанавливает 56 raw bits.
3. Первые 48 бит используются как `base_bits`.
4. Система заново считает HMAC.
5. Сравнивает рассчитанный HMAC с извлеченным `auth_tag`.

Формула:

`auth_ok = extracted_auth_tag == HMAC(secret_key, base_bits)`

Если `true`, payload внутренне согласован.

Если `false`, payload поврежден или извлечен некорректно.

### `payload_document_match`

Тип: boolean или `null`.

Показывает, совпал ли извлеченный hash `document_id` с ожидаемым.

Формула:

`document_match = extracted_document_hash == hash(expected_document_id)`

Если ожидаемый `document_id` не передан, значение равно `null`.

### `payload_corrected_errors`

Тип: integer.

Показывает, сколько одиночных ошибок исправил Hamming-декодер.

Важно: Hamming `(7,4)` может исправить одну ошибку в каждом 7-битном слове. Если ошибок в одном слове больше, исправление может быть неверным.

### `payload_bit_accuracy`

Тип: float или `null`.

Показывает долю совпавших бит между исходным payload и извлеченным payload.

Формула:

`payload_bit_accuracy = N_equal / N_compared`

Если исходные reference bits не переданы, значение равно `null`.

### `payload_ber`

Используется в benchmark.

Показывает долю ошибочных бит payload.

Формула:

`payload_ber = N_error / N_bits`

Связь с `payload_bit_accuracy`:

`payload_ber = 1 - payload_bit_accuracy`

если сравнивается одинаковое количество бит.

### `payload_bpp`

Используется в benchmark.

Показывает, сколько бит payload приходится на один пиксель.

Формула:

`payload_bpp = payload_bits / (H * W)`

Для OmniGuard обычно:

`payload_bits = 100`

## 15. Метрики heatmap и маски

### `tamper_score_mean`

Среднее значение итоговой heatmap.

Формула:

`tamper_score_mean = mean(heatmap_float)`

Интерпретация: общий уровень подозрительности по всему изображению.

### `tamper_score_max`

Максимальное значение heatmap.

Формула:

`tamper_score_max = max(heatmap_float)`

Интерпретация: самая сильная локальная аномалия.

### `tamper_ratio`

Доля пикселей, попавших в бинарную маску.

Формула:

`tamper_ratio = count(binary_mask > 0) / (H * W)`

Пример:

`tamper_ratio = 0.03` означает, что 3% изображения отмечено как подозрительная область.

### `watermark_score_mean`

Среднее значение watermark-map до смешивания с reference-map.

Нужно для диагностики: показывает, насколько сильно сработал именно watermark-источник.

### `watermark_score_max`

Максимальное значение watermark-map.

Нужно для диагностики локального пика по watermark.

### `reference_score_mean`

Среднее значение reference-map.

Есть только если передана защищенная опорная версия.

### `reference_score_max`

Максимальное значение reference-map.

Показывает самый сильный локальный пик различия между текущей и опорной версией.

## 16. Метрики сравнения с reference

Эти метрики появляются в `comparison_metrics`, если передана защищенная опорная версия.

### `mse_vs_reference`

Средняя квадратичная ошибка.

Формула:

`MSE = (1 / (H * W * C)) * sum((R - X)^2)`

где `R` — reference, `X` — текущее изображение.

### `mae_vs_reference`

Средняя абсолютная ошибка.

Формула:

`MAE = (1 / (H * W * C)) * sum(abs(R - X))`

### `rmse_vs_reference`

Корень из MSE.

Формула:

`RMSE = sqrt(MSE)`

### `psnr_vs_reference`

Отношение сигнал/шум между reference и текущим изображением.

Формула:

`PSNR = 10 * log10(255^2 / MSE)`

Чем выше, тем ближе изображения.

### `ssim_vs_reference`

Структурное сходство reference и текущего изображения.

Формула:

`SSIM(x, y) = ((2 * mu_x * mu_y + C1) * (2 * sigma_xy + C2)) / ((mu_x^2 + mu_y^2 + C1) * (sigma_x^2 + sigma_y^2 + C2))`

Чем ближе к `1`, тем больше сходство.

### `changed_pixel_ratio_vs_reference`

Доля пикселей, которые заметно отличаются от reference.

В коде считается максимальная разница по каналам и сравнивается с порогом `12`.

Формула:

`changed = max(abs(R - X), channels) >= 12`

`changed_pixel_ratio = count(changed) / (H * W)`

## 17. Метрики benchmark по маскам

Эти метрики считаются только там, где у атаки есть эталонная маска изменения.

### `mask_precision`

Показывает, какая доля найденной маски действительно совпала с эталоном.

Формула:

`Precision = TP / (TP + FP)`

### `mask_recall`

Показывает, какую долю реальной измененной области система нашла.

Формула:

`Recall = TP / (TP + FN)`

### `mask_f1`

Гармоническое среднее precision и recall.

Формула:

`F1 = 2 * Precision * Recall / (Precision + Recall)`

### `mask_iou`

Intersection over Union.

Формула:

`IoU = intersection(pred_mask, true_mask) / union(pred_mask, true_mask)`

### `mask_dice`

Dice coefficient.

Формула:

`Dice = 2 * intersection(pred_mask, true_mask) / (size(pred_mask) + size(true_mask))`

## 18. Метрики в разделе "Методы встраивания ЦВЗ"

### `PSNR`

Сравнивает исходное изображение и изображение с ЦВЗ.

Используется для оценки незаметности.

### `SSIM`

Сравнивает структуру исходного изображения и изображения с ЦВЗ.

### `bpp`

Показывает емкость встраивания.

Формула:

`bpp = embedded_bits / (H * W)`

### `BER без атаки`

Сравнивает исходные биты ЦВЗ и биты, извлеченные сразу после встраивания.

Если метод работает корректно, значение должно быть близко к `0`.

### `Встроено бит`

Количество бит, которые метод попытался встроить.

Для `LSB` и `DCT` это длина UTF-8 payload в битах.

Для `OmniGuard` это 100 бит encoded payload.

## 19. Метрики пакетного benchmark

Каждая строка benchmark соответствует комбинации:

`image + method + attack`

Колонки:

### `image`

Имя изображения.

### `method`

Метод встраивания: `omniguard`, `lsb`, `dct`.

### `attack`

Название атаки: например `jpeg_q90`, `gaussian_noise_0_01`, `median_3x3`.

### `psnr`

PSNR между исходным изображением и изображением с ЦВЗ.

Важно: в пакетном benchmark это метрика незаметности встраивания, а не сила атаки.

### `ssim`

SSIM между исходным изображением и изображением с ЦВЗ.

### `bpp`

Количество встроенных бит на пиксель.

### `ber`

Доля ошибочных бит после конкретной атаки.

Формула:

`BER = wrong_bits / embedded_bits`

### `embedded_bits`

Сколько бит было встроено.

### `extracted_bits`

Сколько бит было извлечено после атаки.

### `status`

Статус обработки:

- `ok` — строка рассчитана успешно;
- `image_load_error` — изображение не удалось открыть;
- `method_error` — ошибка на уровне метода встраивания или извлечения.

## 20. Атаки в пакетном benchmark

В benchmark используются атаки из требований:

- `jpeg_q100` — JPEG QF=100;
- `jpeg_q90` — JPEG QF=90;
- `jpeg_q50` — JPEG QF=50;
- `gaussian_noise_0_01` — нормальный шум `mean=0`, `sigma=0.01`;
- `salt_pepper` — импульсный шум;
- `random_crop_10` — случайное кадрирование 10%;
- `brightness_plus_15` — прибавить 15 к пикселям;
- `brightness_minus_15` — вычесть 15 из пикселей;
- `brightness_mul_1_5` — умножить яркость на 1.5;
- `brightness_mul_0_8` — умножить яркость на 0.8;
- `median_3x3` — медианный фильтр 3x3;
- `down_up_2x` — уменьшение в 2 раза и обратное увеличение.

Также добавляется строка `no_attack`, чтобы проверить извлечение сразу после встраивания.

## 21. Как работает LSB

LSB заменяет младший бит синего канала.

Формула:

`B'(i,j) = 2 * floor(B(i,j) / 2) + w_k`

Извлечение:

`w'_k = B'(i,j) mod 2`

Плюсы:

- очень простой;
- почти незаметный при малом payload;
- быстро работает.

Минусы:

- плохо переживает JPEG;
- плохо переживает изменение размера;
- плохо переживает фильтры и шум.

## 22. Как работает DCT

DCT работает в частотной области.

Шаги:

1. RGB переводится в `YCrCb`.
2. Берется канал яркости `Y`.
3. Канал делится на блоки `8x8`.
4. Для каждого блока считается DCT.
5. Выбирается коэффициент `(4,3)`.
6. Бит кодируется через четность квантованного коэффициента.

Формулы:

`q = round(C(u,v) / Delta)`

`q' mod 2 = w_k`

`C'(u,v) = q' * Delta`

Извлечение:

`w'_k = round(C'(u,v) / Delta) mod 2`

Плюсы:

- устойчивее LSB к некоторым мягким преобразованиям;
- лучше связан с принципами JPEG.

Минусы:

- может давать больше визуальных искажений;
- зависит от выбранного коэффициента и шага `Delta`;
- не является полноценной защитой от сильных геометрических атак.

## 23. Как читать итоговый вердикт анализа

Вердикт строится не по одному числу, а по нескольким сигналам:

- `payload_auth_ok`;
- `payload_document_match`;
- `tamper_ratio`;
- `tamper_score_max`;
- `changed_pixel_ratio_vs_reference`.

Если payload не прошел HMAC, это сильный признак повреждения встроенных данных.

Если `document_id` не совпал, это признак, что изображение не соответствует ожидаемому идентификатору.

Если `tamper_ratio` заметно больше нуля, система нашла область подозрительных пикселей.

Если `changed_pixel_ratio_vs_reference` заметно больше нуля, текущий файл отличается от защищенной опорной версии.

## 24. Почему heatmap может быть не идеальной

Heatmap — это карта аномалий, а не гарантированная пиксельная разметка.

Она зависит от:

- качества встроенного watermark;
- размера изображения;
- силы атаки;
- наличия reference-изображения;
- выбранного порога;
- сжатия изображения;
- локальных текстур и шумов.

Поэтому для точной локализации лучше использовать режим `hybrid` и передавать защищенную опорную версию.

## 25. Рекомендуемый порядок работы

### Для одного изображения

1. Открыть `Основной сценарий`.
2. Загрузить изображение.
3. Ввести `document_id`.
4. Нажать `Защитить изображение`.
5. Отредактировать область или загрузить внешний измененный файл.
6. Выбрать режим `hybrid`.
7. Запустить анализ.
8. Читать сначала краткую интерпретацию, потом метрики, потом технический отчет.

### Для набора изображений

1. Открыть `Пакетный benchmark`.
2. Загрузить пул файлов или указать папку.
3. Выбрать методы.
4. Ввести payload.
5. Запустить benchmark.
6. Смотреть таблицу и CSV-отчет.

## 26. Главное, что нужно понимать

`payload_auth_ok` отвечает за целостность payload.

`payload_document_match` отвечает за соответствие ожидаемому `document_id`.

`tamper_score_mean` показывает общий фон аномалий.

`tamper_score_max` показывает самый сильный локальный пик.

`tamper_ratio` показывает размер подозрительной области.

`changed_pixel_ratio_vs_reference` показывает физическое отличие от защищенной опорной версии.

`PSNR` и `SSIM` показывают незаметность встраивания.

`bpp` показывает емкость.

`BER` показывает устойчивость извлечения бит после атак.

`Heatmap` показывает не "истину", а распределение подозрительности. Бинарная маска — это результат пороговой обработки heatmap.
